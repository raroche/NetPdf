### Fixed — real-document fidelity batch 3: abspos decorations render, flex-timeline footer overlap, whole-value `var()` border, spurious grid `fr` warning

Five rendering-fidelity fixes root-caused from the travel corpus. Each is byte-identical for every golden suite (goldens, snapshots, corpus, real-doc, and W3C all unchanged; the only behavior deltas are previously-dropped boxes now painting, a footer no longer overlapping, a `border: var(...)` now rendering, and a false-positive diagnostic no longer firing), and each has a paint-proof regression test. Verified against the **local** gate (8489 unit + goldens + AOT + fuzz); the repo's GitHub Actions lanes are currently billing-blocked (jobs fail in ~2 s with no step logs — a runner/billing state, not a code failure), so live CI is red pending that being cleared.

- **Absolutely-positioned `<svg>` / inline "corner" flourishes anchored to a positioned block now render (was dropped ×4).** `11-course-completion-certificate`'s four `.corner` svgs are inline-level abspos boxes that are direct children of a `position:relative` frame which also has block children. `BoxBuilder.FixupAnonymousBlocks` was sweeping them into an anonymous inline-only block (reusing the frame's `position:relative` style); that wrapper is emitted through the inline-only path, which never records positioned-box geometry, so the corners' containing block couldn't resolve and all four were dropped (`LAYOUT-ABSOLUTE-FEATURE-UNSUPPORTED-001`). Out-of-flow boxes are now **excluded from the anonymous-block run-wrapping** (CSS 2.1 §9.2.4 — they don't participate in the inline/block split), so they stay direct children of the real positioned parent (whose geometry *is* recorded). Closes `deferrals.md` RC2 residual (4).
- **`position:absolute` boxes anchored to a positioned block *descendant inside a flex item* now render (was dropped).** The common real shape — a bullet inside a `position:relative` `<li>` in a flex-item `<ul>` (`.opt ul{flex:1}` in `02-travel-quote`). The `<li>`'s geometry is emitted through the flex item's content buffer, which did not record positioned-box geometry, so the descendant's containing block was unresolved and the box dropped. `BufferingMeasureSink.FlushTo` now records positioned-CB geometry for every CB-establishing block descendant it flushes, and the redundant nested item-content abspos pass — which only ever dropped the descendant and leaked a spurious `LAYOUT-ABSOLUTE-FEATURE-UNSUPPORTED-001` (FlexLayouter is not an abspos delegation boundary; the top-level pass owns flex-item abspos) — is suppressed via a new `suppressOutOfFlowEmission` flag. Real `02` went from **34 abspos drops to 0**. Closes `deferrals.md` RC2 residual (1).
- **A footer after an auto-height flex "timeline" no longer overlaps the flex content.** An auto-height row flex's used cross (block) size is its content extent, but `FlexLinePacker` sizes each line from the items' *declared* cross only (0 for a content-determined item — it never content-measures), so the container reported a chrome-only block box. A block stacking such flex rows (`03-itinerary`'s `.timeline` of `.day` rows) under-advanced its cursor and the trailing `.note` overlapped the day content. `FlexLayouter` now folds the (already-computed) per-item content cross into `LastEmittedBlockExtent`, and both flex-dispatch resize consumers resize the wrapper + advance the sibling cursor by it for a non-paginating auto-height row flex (previously only paginating wrappers were resized); the pagination measure folds the extent in symmetrically. The note now lands below the timeline (`AutoHeightFlexTimelineFooterTests`). *Known follow-up:* with the timeline now measured tall, a large such stack still paginates conservatively (atomic-subtree granularity — the deferred "mid-split" work), so `03` renders on more pages than strictly necessary; the overlap (the visible defect) is resolved.
- **A whole-value `var()` `border` shorthand (`border: var(--rule)`) now renders.** AngleSharp.Css drops it, and it can't be split into width/style/color before substitution (so the "emit each longhand with the same var" recovery that works for `background: var(--x)` fails — `border`'s resolved value is multi-token). `CssPreprocessor` now recovers the *shorthand itself* so it reaches the cascade, and `VarResolver` expands the `border[-side]` shorthand into its longhands *after* substituting the `var()` — the cascade's only shorthand-expansion site, firing solely for a whole-value-var border (byte-identical otherwise). Replaces the prior "skip, render nothing" behavior. `Whole_value_var_border_shorthand_renders_after_substitution`.
- **A spurious `LAYOUT-GRID-FR-UNDER-INDEFINITE-APPROXIMATED-001` warning on definite-width auto grids is suppressed.** The diagnostic keyed purely on the `width`/`height` slot being `auto`, but an auto-`width` grid nested in a definite containing chain (`repeat(4,1fr)` columns inside a flex item — `10-event-ticket`'s `.info-grid`) has a *definite* used inline size per CSS Grid §5.2, so its `fr` columns resolve exactly. It now fires only when the axis extent is genuinely indefinite; the block axis (auto-height `fr` rows sized against the fragmentainer fallback) still warns. PDF-byte-identical — only the diagnostic stream changes.

### Fixed — real-document fidelity: `var()` shorthands, flex percentage sizing, padded-box over-inflation

A batch of rendering-fidelity fixes root-caused from a visual review of a 12-document travel corpus. Each is byte-identical for every golden suite (gated to the specific pattern), and each has a paint-proof regression test.

- **`var()` inside a `background` / `border` shorthand now renders (was silently dropped).** AngleSharp.Css rejects a shorthand whose value contains `var()` (it can't validate the reference before substitution), so `background: linear-gradient(…, var(--a), var(--b))`, `background: var(--c)`, and `border-bottom: 1px solid var(--d)` produced *nothing* — with no diagnostic. `CssPreprocessor` now recovers such a shorthand by re-expanding it to its longhands with the `var()` intact. A `background` shorthand emits **both** `background-image` and `background-color` (the unset one at its initial `none` / `transparent`) so a later `background: var(--color)` correctly resets a prior `background-image`; `border[-side]` expands to `-width` / `-style` / `-color` with validation deferred to `VarResolver`. A whole-value `var()` (`border: var(--rule)`) is left unrecovered (it can't be classified before substitution) rather than misexpanded. Fixes bar-chart fills, logos, and signature rules that used a `var()`-carrying shorthand.
- **A percentage *main* size on a flex item resolves against the container.** A flex item with `width: 48%` (or `height:%` on a column) and the default `flex-basis: auto` collapsed to 0 — only a `LengthPx` was honored — so `justify-content: space-between` shoved the second column off the page edge. It now resolves the percentage against the container's definite main size (mirroring the `flex-basis` percentage branch). Follows PR #271, which fixed the `width: auto` case.
- **A percentage *cross* size on a flex item resolves against a definite-cross-size container.** A bar with `height: 100%` in a `height: 58px` row-flex collapsed to zero height and painted no rectangle. A new `CrossBorderBoxSizePx` overload resolves the percentage against the container's definite cross extent, threaded through the line packer and emission. An indefinite container cross size still yields 0 (auto against an indefinite parent), so auto-height flex is unchanged.
- **A padded auto-height box measured with pretty-printed children no longer over-inflates.** The subtree-visual-extent measure charged inherited padding/border to whitespace `AnonymousBlock` wrappers (which reuse the parent's style) and counted out-of-flow children — while the painter draws neither — so a padded container whose block children were separated by source whitespace ballooned by roughly its padding per gap. The measure now skips out-of-flow children and treats an anonymous box's own chrome as zero, matching the painter.
- **A nested flex row split across a page boundary now resumes on the next page instead of re-rendering wholly.** On a fresh resume page (with spare room) the would-overflow pagination gate stayed off, so a resumed flex ignored its continuation's resume-cut, re-emitted the whole item (duplicated content), and under-advanced the sibling cursor — so a trailing block painted over the flex content. A flex child resuming a dual-input split now activates the pagination path regardless of the overflow gate. *Residual:* the `03-itinerary` corpus footer overlap is a **separate** bug this does not resolve — an auto-height, non-paginatable flex container is measured as its chrome-only border box (its flex content height ignored) in the pagination subtree measure, so a stack of such flex rows under-measures and its trailing sibling overlaps. Precisely root-caused; the fix (measure the flex content height) changes the pagination measure and is tracked in `docs/deferrals.md`.

### Fixed — `position: absolute` boxes anchored to a positioned flex item now render (RC2, partial)

- A `position: absolute` box whose containing block is a **`position: relative` flex item** (the flex item is its nearest positioned ancestor — including when the box is nested under static descendants of the item, not only a direct child) was being dropped with `LAYOUT-ABSOLUTE-FEATURE-UNSUPPORTED-001`. Unlike a grid item (which `GridLayouter` lays out via a nested `BlockLayouter` that owns its own out-of-flow pass) or a flex/grid **container** (a block-level box emitted through the block-flow path that records its geometry), a flex **item** is emitted by `FlexLayouter` directly — and its border-box geometry was never recorded, so the abspos pass couldn't find the item as a containing block and deferred (dropped) the descendant. `FlexLayouter` now records each emitted item's geometry through a callback into the dispatching `BlockLayouter`, so the box anchors to the item and paints.
- **Scope (honest):** this fixes only the case where the abspos box's containing block **is** the flex item. It does **not** yet cover the common real-world shape where the abspos box anchors to a `position: relative` **block descendant nested inside** a flex item — e.g. a bullet `li::before` whose containing block is a positioned `<li>` inside a flex-item `<ul>`. That descendant's geometry is emitted through the flex item's content buffer, which does not record it, so such boxes are still deferred. Tracked in `deferrals.md#abspos-cycle-1-explicit-only` as a follow-up.
- Byte-identical for every golden suite (no golden fixture had an abspos descendant of a positioned flex item; the only behavior change is that a previously-dropped box now renders). Remaining edges: (1) an abspos descendant of a *page-sliced* flex item stays deferred, consistent with abspos-pagination; (2) the nested-descendant shape above.

### Fixed — CSS-wide keywords on paint properties no longer emit spurious "unsupported" diagnostics

The paint properties are read as **raw** cascade winners (bypassing the typed resolver), so a CSS-wide keyword (`initial` / `inherit` / `unset` / `revert`) reached the value parsers verbatim and was rejected as unsupported — firing a spurious diagnostic on nearly every document (the extremely common `background: #color` shorthand expands to `background-image: initial`). All of these properties are non-inherited, so `initial` and `unset` are exactly the reset (`none` / no effect); `inherit` and `revert` are *approximated* to the same reset here (the raw-read path doesn't resolve cross-origin or parent-computed values — a rare case, and the reset is the near-universal result). Either way the keyword is not itself a paint value, so it's now treated as the reset with no diagnostic. Extends the central CSS-wide-keyword handling (from the `PropertyResolverDispatch` fix) to the raw-read paint path across:

- **`background-image`** (single-layer) — `background: #color` no longer warns.
- **`box-shadow`** + **`text-shadow`**.
- **`transform`**, **`filter`**, **`clip-path`**.
- **`border-image`** and the element-level **`filter`** / **`mask`** references (the shared `<img>` / inline-`<svg>` image-spec path included).

Byte-identical output (a CSS-wide keyword already produced the same no-effect result — only the misleading diagnostic is gone). Tests: `CssWideKeywordPaintTests`, with a guard that a real gradient still renders.

### Fixed — Rendering fidelity: flex intrinsic over-measurement + inline `<svg>` rendering

- **Flex over-measurement collapse (RC1)** — a flex item whose content was block-level children measured at the huge max-content available width (a `width:auto` block filled it), so its flex base size blew up, the flex line falsely overflowed, and the default `flex-shrink` collapsed the *sibling* toward min-content — the "header wraps one word per line", card, and label/value-collision failures. `BufferingMeasureSink` now contributes a block fragment's box width to the intrinsic content extent only when it has a **definite** inline size (an explicit absolute width, a replaced element's intrinsic size, or an explicit length `min-width` floor); an auto/`%`-width block's extent comes from its descendants.
- **Inline `<svg>` rendering (RC3)** — an inline `<svg>` element in HTML flow is now a **replaced element** (sized from its `width`/`height`/`viewBox`), and its own markup renders through the existing SVG pipeline (rasterizer + SEC-8 security guards), content-keyed so repeated logos dedup. `currentColor` is seeded from the element's computed `color` before rasterization, and it reuses the `<img>` image-spec path so `object-position` / `filter` / `mask` apply. Previously an inline `<svg>` was an unhandled block that filled the available width (the concrete trigger of the flex collapse above) and drew nothing — so logos, seals, crests, and QR codes were blank.
- Byte-identical for every golden suite (no golden fixture had an inline `<svg>` element or a nested-block-in-flex max-content path that changed).

### Added — Document navigation: internal `#fragment` links + catalog initial view

- **Same-document links** — `<a href="#id">` now emits an internal `/GoTo` `/Link` annotation that jumps to the target element's page + position (a `/Dest [page /XYZ null top null]`). Resolution is document-wide and two-phase, so a link can target an element on a *later* page. The `#id` fragment is UTF-8 percent-decoded (browser fragment-navigation semantics), so `href="#r%C3%A9sum%C3%A9"` matches `id="résumé"` and `href="#sec%2F2"` matches `id="sec/2"`. Block, inline-block, **and inline-flow** targets all resolve (an inline `<span id>` anchors to its containing block/line position). A dangling target is reported once as `LINK-FRAGMENT-UNRESOLVED-001` and the link text still renders. On the link side, the anchor must produce its own box fragment (block / inline-block); inline-flow anchors remain the documented shared follow-up.
- **Initial view** — `HtmlPdfOptions.PageLayout` (`PdfPageLayout`: SinglePage / OneColumn / TwoColumn{Left,Right} / TwoPage{Left,Right}) and `PageMode` (`PdfPageMode`: UseNone / UseOutlines / UseThumbs / FullScreen / UseOC / UseAttachments) set the catalog `/PageLayout` and `/PageMode` — e.g. open a report with the bookmarks panel showing. Both default to omitted, so a document that requests no initial view is byte-for-byte unchanged.

### Added — Document metadata: HTML-driven `/Info`, XMP `/Metadata`, catalog `/Lang` + `/ViewerPreferences`

- **HTML head → PDF metadata** — the document `<title>`, `<meta name="author|description|keywords">`, and root `<html lang>` are now harvested and flow into the PDF `/Info` dictionary (`/Title`, `/Author`, `/Subject`, `/Keywords`) and the catalog `/Lang`. Previously metadata was set only via options and never read from the HTML, so generated PDFs were catalogued by filename rather than by name.
- **Options precedence + new fields** — `HtmlPdfOptions` gains `Subject`, `Keywords`, `Creator`, `CreationDate`, `ModDate`, and `DocumentProperties`. An explicit option overrides the HTML-harvested value; `Language` (now defaulting to `null` rather than `"en"`, so a document is no longer presumed English) is the fallback when `<html lang>` is absent. Dates default to omitted, preserving the deterministic-by-default output.
- **XMP metadata stream** — a deterministic XMP (`/Type /Metadata /Subtype /XML`) packet mirroring the descriptive fields (Dublin Core `dc:title`/`dc:creator`/`dc:description`/`dc:language` + `pdf:Keywords`/`pdf:Producer` + `xmp:CreatorTool`) and, when set, the dates (`xmp:CreateDate`/`xmp:ModifyDate`) is emitted whenever any metadata is present — the modern-reader/archival carrier and the prerequisite for a later PDF/A pass. XMP shares the exact same sanitize + `MaxMetadataChars` length cap as `/Info`, so the two can't disagree and hostile input can't inflate the stream.
- **Viewer preferences** — a document with a title emits `/ViewerPreferences << /DisplayDocTitle true >>` so readers show the document title in their chrome instead of the filename.
- **Custom `/Info` entries** — `DocumentProperties` writes arbitrary extra keys into `/Info` (reserved standard keys + `Producer` cannot be shadowed; non-emittable keys are skipped rather than crashing). Entries emit in `StringComparer.Ordinal` key order — not the caller's dictionary iteration order — so output stays byte-stable across `IReadOnlyDictionary` implementations.
- **Byte-stable for bare documents** — a document with no head metadata emits none of the new catalog entries, so existing output is unchanged; the four metadata-bearing determinism snapshots were re-pinned for the added `/ViewerPreferences` + XMP stream.

### Added — Phase 5 (packaging): layout `lang` routing + the full language-pack set (work-breakdown tasks routing + 6–9)

- **Layout hyphenation now routes by a block's effective HTML `lang`.** `HyphenationRegistry` moved from the `NetPdf` facade into `NetPdf.Text` (public namespace `NetPdf.Hyphenation` unchanged) so `NetPdf.Layout` can consult it. `BlockLayouter` resolves the `hyphens: auto` hyphenator from a block's effective `lang` — the nearest `lang` attribute up the ancestor chain (e.g. `<html lang="de">`) — via `HyphenationRegistry.ResolveOrDefault`. **Byte-identical to the prior behavior** unless a `NetPdf.Languages.*` pack has registered the language (`ResolveOrDefault` returns the bundled English hyphenator for an unregistered language); the shaping language argument is unchanged. This makes the language packs actually affect rendered output — a `<html lang="de">` document with the European pack loaded now hyphenates with German rules. Per-run/mixed-language `lang` and shaping-language routing remain follow-ups. The `NetPdf.Languages.European` pack now references `NetPdf.Text` compile-only (`PrivateAssets="all"`) for the relocated type while keeping its single `NetPdf` package dependency.
- **`HyphenationRegistry.RegisterNoHyphenation(language)`** — a new public method that registers a language as one that does NOT soft-hyphenate (a no-op hyphenator), so `hyphens: auto` yields no breaks for it instead of the English fallback.
- **`NetPdf.Languages.Cjk`** — registers Chinese/Japanese/Korean (`zh`/`ja`/`ko`) as no-hyphenation languages (CJK line breaking is per-character, UAX #14). Richer CJK line-break tailoring (kinsoku) is a follow-up.
- **`NetPdf.Languages.Arabic`** — registers Arabic/Persian/Urdu (`ar`/`fa`/`ur`) as no-hyphenation languages (Arabic-script justification uses kashida, not hyphens).
- **`NetPdf.Languages.Indic`** — registers the major Indic-script languages (Hindi, Sanskrit, Marathi, Nepali, Bengali, Assamese, Panjabi, Gujarati, Odia, Tamil, Telugu, Kannada, Malayalam) as routing-aware placeholders. Indic scripts *do* hyphenate, but this build ships no pattern data yet — the CTAN `hyph-utf8` (LPPL) data is maintainer-vendored; until then `hyphens: auto` produces no breaks for them (conservative — never the wrong English hyphenation).
- **`NetPdf.Languages.All`** — a convenience meta-package depending on the four leaf packs; `AllLanguages.Register()` registers them all in one call. Each leaf pack's produced nuspec declares a single `NetPdf` dependency; `.All` depends on exactly the four leaf packs (pinned by `LanguagePackShapeTests`).

### Security — Phase 5 hardening sub-track: SEC-2 (regression suite + fuzz foundation)

- **Consolidated security regression suite** — `tests/NetPdf.UnitTests/Security/SecurityHardeningTests.cs` (`Category=Security`, 60 test cases at closeout across the security track) pins the SSRF / LFI / XXE / DoS / malicious-PDF-output invariants as executable assertions, organized by the threat-model taxonomy (`docs/security/security-hardening-plan.md`). ### Security — SEC-3: SSRF choke-point invariant (the sequencing gate before Phase-5 resource loading)

- **Enforced choke-point invariant** — `tests/NetPdf.UnitTests/Security/ResourceChokePointInvariantTests.cs` scans the production sources and fails the build if any code outside `SafeResourceLoader` invokes the raw `IResourceLoader.LoadAsync`, or constructs an `HttpClient`/`SocketsHttpHandler` outside `SafeHttpResourceLoader`. This turns the documented "every resource fetch goes through the one wrapper" promise into a CI guard, so the Phase-5 CSS/font-loading feature can't silently reintroduce SSRF (the #1 HTML-to-PDF bug class).
- **Now-passing negative + SSRF + MIME tests** (in the `SecurityHardeningTests` SEC-3 region): `@import` / `@font-face src:url()` / `<link rel=stylesheet>` make **zero fetches** under `UntrustedHtml` (and under an http-permissive policy — a tripwire for when the fetch is wired); `SafeResourceLoader.FetchAsync` blocks the cloud-metadata IP for the `Stylesheet` and `Font` resource kinds (loader never invoked); MIME mismatch is rejected for those kinds (`text/html` served as a stylesheet/font is refused, `text/css`/`font/ttf` accepted); the kind→MIME allowlist matrix is pinned and unknown resource kinds are **fail-closed** (fetch nothing). Test-only change — no engine behavior change. When Phase 5 wires CSS/font fetching it must route through `SafeResourceLoader.FetchAsync(…, Stylesheet|Font)`; these guards then protect that path.

### Security — SEC-4: DNS-resolution timeout (slow-resolver DoS)

- **`SafeHttpResourceLoader` now bounds host resolution** at `min(ResourceTimeout, 5s)` via `Task.WaitAsync`. `getaddrinfo` does not reliably honor the cancellation token on every platform, so an unbounded resolve of a dead or hostile-DNS host could hang the render thread. On timeout it throws a typed `HttpRequestException` ("DNS resolution … timed out after Ns"), which `SafeResourceLoader` catches and surfaces as a typed resource-failure — fail-fast, not a hung thread. The host resolver is now injectable through an internal constructor for testing. Only affects the opt-in `SafeHttpResourceLoader` (no default loader ships), so all render output is byte-identical; the golden/corpus/snapshot gates are unchanged.

### Security — SEC-5: configurable output-byte + page-count caps

- **`SecurityPolicy` gains `MaxPages` and `MaxOutputBytes`.** Defaults preserve trusted behavior (`MaxPages` = 20 000, the hard backstop; `MaxOutputBytes` effectively unlimited); `UntrustedHtml` tightens both to 500 pages / 50 MiB. The render pipeline enforces the configured page cap in the multi-page loop (above the renamed `MaxPagesBackstop` = 20 000 emergency guard) → stops and emits `PDF-PAGE-LIMIT-EXCEEDED-001`; and checks the produced size after `Save()` → aborts with `HtmlPdfException(PDF-OUTPUT-SIZE-EXCEEDED-001)` (a truncated PDF isn't meaningful). Byte-identical for any document that doesn't trip the caps.

### Security — SEC-6: data: URI MIME hardening

- **`data:` URIs now require an explicit, allowlisted mediatype** at the resource choke point. A missing/empty type (`data:,…` / `data:;base64,…`) or a non-allowlisted one (esp. `text/html`, an SSRF/polyglot sneak-path) is rejected. The HTML `<img>` path already enforced this; SEC-6 closes it for CSS `url()` / direct loader use. `data:` stays fully off under `UntrustedHtml`.

### Security — SEC-7: WOFF/WOFF2 decompression guard + re-validation

- **A 100:1 decompression-ratio cap** (above a 1 MiB floor) added to both the WOFF and WOFF2 decoders, complementing the existing absolute-size caps, and the **reconstructed sfnt is re-run through `FontSafetyValidator`** before it could reach a shaper. Guards the (still dormant, v1-rejected) WOFF render wiring against a decompression bomb. `FontFaceData`'s doc already reflected the rejected-in-v1 state.

### Security — SEC-8: SVG parser observability

- **SVG parse failures are now diagnosed, not silent.** `SvgRasterizer.TryRender` reports an `SvgParseStatus`; the pipeline emits `SVG-PARSE-FAILED-001` distinguishing a security-guard rejection (a prohibited `<!DOCTYPE`/`<!ENTITY` or an over-size document — the XXE / entity-expansion defense) from malformed XML, instead of the prior silent null / misleading generic image-decode error. Valid-SVG rendering is byte-identical.

### Security — SEC-9: CSS-amplification caps (verified + regression-tested)

- Confirmed and regression-tested the existing CSS-amplification guards (the audit gap was largely stale): grid `repeat()` is capped at 10 000 (over-cap → `CSS-PROPERTY-VALUE-INVALID-001` + the declaration dropped), the expanded track count at 50 000, and counter magnitude is bounded by the counter formatters (roman > 3999 → decimal; alphabetic/greek logarithmic).

- **`border-image-width`** (§6.4): the dest thickness of each edge/corner band — `<length>` / `<number>` (× the border width) / `<percentage>` (of the border-image area) / `auto` (the intrinsic slice size). Overlapping widths are reduced proportionally per axis. Defaults to `1` (= the border width).
- **`border-image-outset`** (§6.5): grows the border-image area outward from the border box by a `<length>` / `<number>` (× the border width). Defaults to `0`.
- The prior "approximated sub-feature" diagnostic (`CSS-BORDER-IMAGE-UNSUPPORTED-001`) no longer fires for repeat/width/outset; it now only flags a non-`url()` (gradient) source. The `fill` keyword center is still stretched (a minor residual).

### Added — Phase 4 visual parity: CSS filters on images (PR 2)

`filter` now applies to `<img>` elements via a Skia raster fallback (PDF has no native filter primitive). Five tasks; gated so unfiltered images stay byte-identical (all 6 byte-identity gates + AOT sha256 `2942DD1E…` unchanged).

- **The ten CSS Filter Effects L1 functions** — `grayscale` / `sepia` / `invert` / `brightness` / `contrast` / `saturate` / `hue-rotate` / `opacity` (color matrices, §C; omitted amount defaults to 1, and grayscale/invert/sepia/opacity clamp above 1), `blur` (Gaussian; the length is the σ directly), and `drop-shadow` (alpha-following; the blur length is σ directly, like other filters — not 2σ). A new `CssFilter_Parser`, the Skia `ImageFilterApplier` (a composed `SKImageFilter` chain applied to the decoded image, re-embedded as a raster XObject + alpha `/SMask`), and `ImageFilterBridge` (CssFilter → applier steps, resolving the drop-shadow color). Chained filters compose in declaration order. Source dimensions are preflighted via `SKCodec` (header only) so an oversized image is rejected before a full pixel decode. The filtered XObject is deduped by the RESOLVED steps (incl. the shadow RGBA, so a `currentColor` drop-shadow doesn't collide across elements). `CSS-FILTER-RASTER-FALLBACK-001` (Info); an unparseable value (`url(#id)`, bad color) surfaces `CSS-FILTER-UNSUPPORTED-001` (Warning).
- **`drop-shadow` expands the raster bounds** — the shadow extends outside the image box, so the applier pads the frame by the shadow extent and returns `FilterPadding` (fractions of the source size); the painter expands the image placement so the content stays box-aligned and the shadow extends outward. Per-`(image+filter)` deduped.
- **`filter` is read as the box's raw cascade winner** (recovered via `CssPreprocessor` — AngleSharp.Css drops it, like `transform`); the source bytes are retained on the image cache `Entry` so the filtered variant can be produced.
- **A `filter` on a NON-replaced element** (a `<div>` / text box) is deferred with `CSS-FILTER-ELEMENT-UNSUPPORTED-001` (Warning) and paints unfiltered — filtering a general element's rendered subtree needs a Skia subtree renderer NetPdf doesn't have yet (a tracked follow-up; the painter draws straight to PDF). Filters on images ARE applied.
- Residuals (documented in `deferrals.md`): general element-subtree filters (the subtree renderer), blur/drop-shadow lengths applied in the image's INTRINSIC pixel space (exact at ~intrinsic display size), the blur halo clipped at the image box, filters on background images, and `url()` SVG-filter references.

### Added — Phase 4 visual parity: gradient + shadow refinements (PR 1)

Finishes the #209/#210 residuals — five tasks, each gated so non-gradient / non-shadow rendering stays byte-identical (all 6 byte-identity gates + AOT sha256 `2942DD1E…` unchanged).

- **`conic-gradient(...)` / `repeating-conic-gradient(...)`** now render via the Skia raster bridge (PDF has no native conic shading): a new `CssConicGradient` parser (`[from <angle>]? [at <position>]?` + angular color stops in deg/grad/rad/turn/%) and `ConicGradientRasterizer` (an `SKShader.CreateSweepGradient` rotated so CSS 0° = up clockwise) → an image XObject with an alpha `/SMask`, emitting `CSS-CONIC-GRADIENT-RASTER-001`. AngleSharp.Css drops the `repeating-*-gradient` value forms, so `CssPreprocessor` now value-gates a `background-image` recovery for them.
- **Length-positioned gradient color stops** (`linear-gradient(..., blue 50px, ...)`) — `px` + the absolute units (pt/pc/in/cm/mm/Q) resolve against the gradient-line length (CSS Images §3.1); em/rem/% still reject (no font/CB context). Byte-equivalent to the matching `%` form.
- **`repeating-linear-gradient` / `repeating-radial-gradient`** render as NATIVE PDF shadings (no raster): the `last − first` specified-stop period is tiled across the gradient line (prior + next cycles covering [0,1], so a non-zero first stop repeats correctly; a zero-width period collapses to the average color; bounded) so the axial/radial shading repeats. A full-turn period ≡ the plain gradient. (Residual: a repeating-radial under a `closest-*` extent clamps beyond the ending shape; the default farthest-corner is exact.)
- **Out-of-range color stops** (`red -50px`, `blue 150%`, conic `white -180deg`/`black 540deg`) are kept raw through the shared stop pipeline and clip to the visible [0,1] line with interpolated boundary colors, rather than being clamped at parse time. Linear/radial/conic share one `ResolveAndNormalizeStops` (§3.4 defaults, non-decreasing fixup, repeating tiling, unit-interval clip).
- **Radial-gradient elliptical shaping** — the default (ellipse) shape on a non-square box now renders a true ellipse: the per-axis ending radii (CSS Images §3.2.1) are realized by registering a circle of the larger radius and squashing the other axis with a CTM scale about the center (a `cm` between the clip and the `sh`, new `PaintShadingInRect` param). A circle, or a centered ellipse on a square box (rx == ry), emits no CTM — byte-identical with the pre-ellipse output.
- **`box-shadow: inset`** now paints (was skipped): inset layers draw OVER the background, clipped to the padding box, UNDER the border — a native even-odd ring (sharp) or a Skia-rasterized band (blurred, via `ShadowRasterizer.TryRasterizeInset` — fill + a Gaussian-blurred `DstOut` hole). Outset + inset compose in one value.
- Remaining gradient/shadow residuals (documented in `deferrals.md`): per-stop alpha on the NATIVE linear/radial shadings (a soft-mask), color-interpolation hints, multi-layer background lists, gradient `background-clip`/`-origin` insets, text-shadow blur/inheritance, transform 3D-projection + stacking context.

### Added — Phase 4 visual parity: shadows + 2D transforms

- **`box-shadow`** (outset) now paints UNDER the background (CSS B&B §7.2): a sharp (blur ≈ 0) layer is a native filled (rounded) rect offset + spread-expanded from the border box (corner radii grown by the spread); a blurred layer rasters through a new Skia bridge (`ShadowRasterizer` — draws the rounded-rect shape, applies a Gaussian blur with σ = blur/2, and places it as an image XObject with an alpha `/SMask`) at 2× resolution, capped at 4096 px/side, emitting `CSS-BOXSHADOW-BLUR-RASTER-001`.
